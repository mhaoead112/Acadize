## 1. Project Overview

### What This Application Is
Eduverse is a comprehensive **Learning Management System (LMS)** built as a full-stack web application. It provides a complete educational platform for online learning, course management, assessments, and communication.

### Who It Is For
- **Students**: Access courses, complete assignments, take exams, track progress, participate in study groups
- **Teachers**: Create courses, manage content, grade assignments, monitor student performance, communicate with students and parents
- **Admins**: Manage users, oversee system-wide settings, create courses, manage calendar events, handle announcements
- **Parents**: Monitor their children's academic progress, grades, attendance, communicate with teachers

### Core Problem It Solves
Eduverse consolidates all aspects of educational delivery and management into a single platform:
- Course content delivery and organization
- Assignment submission and grading
- Secure exam administration with anti-cheat monitoring
- Real-time communication (messaging, study groups)
- Progress tracking and analytics
- Parent-teacher-student collaboration

### High-Level Feature Set
1. **Course Management**: Course creation, lesson uploads (PDF, DOCX, video), enrollment management
2. **Assignment System**: Assignment creation, file submissions, grading with feedback
3. **Exam System**: Comprehensive exam creation with multiple question types, anti-cheat monitoring, adaptive retakes
4. **Messaging**: Direct messages, study groups, parent-teacher communication
5. **Progress Tracking**: Student dashboards, analytics, grade reports, report cards
6. **Calendar & Events**: Course events, meetings, announcements
7. **AI Study Buddy**: Educational chatbot with multiple personality modes
8. **Anti-Cheat System**: Tab switching detection, webcam monitoring, integrity scoring
9. **Mistake Tracking**: Identifies student mistakes, generates adaptive retake exams
10. **Parent Portal**: Child monitoring, grade viewing, teacher communication

### Separation of Concerns
- **Client (`client/src/`)**: Presentation layer, user interactions, routing
- **Server (`server/src/`)**: Business logic, data validation, API endpoints
- **Database (`server/src/db/schema.ts`)**: Data persistence, relationships, constraints
- **Shared (`shared/`)**: Type definitions, validation schemas shared between client/server

### Key Architectural Patterns
1. **Role-Based Access Control (RBAC)**: All routes and components enforce role-based permissions
2. **Service Layer Pattern**: Business logic isolated in `server/src/services/`
3. **Repository Pattern**: Database access abstracted through `storage.ts`
4. **API Route Modules**: Each feature has dedicated route file in `server/src/api/`
5. **React Query for Server State**: All API calls managed through React Query hooks
6. **Context Providers**: Auth, Theme, Notifications managed via React Context

### Where Business Logic Lives
- **Server-Side**: `server/src/services/` contains all core business logic
  - `exam-attempt.service.ts`: Exam lifecycle, grading, submission
  - `anti-cheat-monitor.service.ts`: Integrity monitoring, event processing
  - `risk-scoring.service.ts`: Risk assessment algorithms
  - `retake-exam-generator.service.ts`: Adaptive retake generation
  - `answer-evaluation.service.ts`: Answer grading logic
  - `parent.service.ts`: Parent portal data aggregation
  - `admin.service.ts`: Admin operations, user management

### Where UI Logic Lives
- **Client-Side**: `client/src/pages/` and `client/src/components/`
  - Pages are role-specific (student-*, teacher-*, admin-*, parent-*)
  - Shared components in `client/src/components/`
  - Custom hooks in `client/src/hooks/`

## 4. Folder & Module Responsibilities

### `client/src/`
**Purpose**: Frontend application code
- **`pages/`**: 124 page components, organized by role (student-*, teacher-*, admin-*, parent-*)
  - **Add**: New pages for new features, following naming convention `{role}-{feature}.tsx`
  - **Do NOT add**: Business logic, direct database access, API implementations

  - **Add**: Shared UI components, layout components, form elements
  - **Do NOT add**: Page-specific logic, API calls (use hooks instead)

  - **Add**: Reusable stateful logic, API query hooks
  - **Dependency rules**: Can import from `lib/`, cannot import from `pages/`
- **`contexts/`**: React Context providers (Auth, Theme, Notifications)
  - **Add**: New global state providers sparingly
  - **Do NOT add**: Feature-specific state (use React Query or local state)
- **`lib/`**: Utility functions, API client, query client setup
  - **Add**: Pure utility functions, API client configuration
  - **Do NOT add**: Component logic, stateful code

### `server/src/`
**Purpose**: Backend API and business logic
- **`api/`**: 37 route modules, one per feature domain
  - **Add**: New route files for new features, following pattern `{feature}.routes.ts`
  - **Dependency rules**: Can import from `services/`, `db/`, `middleware/`
  - **Do NOT add**: Business logic (belongs in `services/`), direct database queries (use services)
- **`services/`**: 25 service modules containing business logic
  - **Add**: New service files for complex business logic
  - **Dependency rules**: Can import from `db/`, other services
  - **Do NOT add**: HTTP request/response handling (belongs in routes)
- **`db/`**: Database schema and connection
  - **`schema.ts`**: 40+ table definitions with Drizzle ORM
  - **Add**: New table definitions, enums, indexes
  - **Do NOT add**: Business logic, API endpoints
- **`middleware/`**: Express middleware (auth, error handling)
  - **Add**: Cross-cutting concerns (logging, validation)
  - **Do NOT add**: Feature-specific logic

### `shared/`
**Purpose**: Code shared between client and server
- **Add**: Type definitions, validation schemas (Zod)
- **Do NOT add**: Implementation code, framework-specific code

### `migrations/`
**Purpose**: Database migration files
- **Add**: New migration files via `npm run db:push` or Drizzle Kit
- **Do NOT modify**: Existing migration files (create new ones instead)

## 5. Routing & Navigation

### How Routing Is Implemented
- **Library**: Wouter (lightweight alternative to React Router)
- **Location**: `client/src/App.tsx` contains all route definitions
- **Pattern**: Declarative route definitions with `<Route>` components
- **Protection**: Route wrapper components enforce authentication and role-based access

### Student vs Teacher vs Admin Route Separation
Routes are strictly separated by role using wrapper components:

```typescript
// Student routes: /student/*
<StudentRoute>
  <StudentDashboard />
</StudentRoute>

// Teacher routes: /teacher/*
<TeacherRoute>
  <TeacherDashboard />
</TeacherRoute>

// Admin routes: /admin/*
<AdminRoute>
  <AdminDashboard />
</AdminRoute>

// Parent routes: /parent/*
<ParentRoute>
  <ParentDashboard />
</ParentRoute>
```

**Route Prefixes**:
- `/student/*` - Student-only pages
- `/teacher/*` - Teacher-only pages (teachers and admins)
- `/admin/*` - Admin-only pages
- `/parent/*` - Parent-only pages
- `/` - Public landing pages (no auth required)

### Navigation Guards and Constraints
**Protection Components** (`client/src/components/ProtectedRoute.tsx`):
1. **`PublicOnlyRoute`**: Redirects authenticated users to their role dashboard
2. **`StudentRoute`**: Requires `role === 'student'`, redirects to `/login` if not authenticated
3. **`TeacherRoute`**: Requires `role === 'teacher' || role === 'admin'`
4. **`AdminRoute`**: Requires `role === 'admin'`
5. **`ParentRoute`**: Requires `role === 'parent'`
6. **`MultiRoleRoute`**: Accepts array of allowed roles
7. **`ProtectedRoute`**: Generic authenticated route (any role)

**Server-Side Guards** (`server/src/middleware/auth.middleware.ts`):
- `isAuthenticated`: Verifies JWT token
- `requireRole(['teacher', 'admin'])`: Enforces role-based access
- Middleware applied to all protected API routes

### Exam Lock / Protected Flows
**Exam Attempt Flow** (CRITICAL - DO NOT MODIFY LIGHTLY):
1. `/student/exams` - List available exams
2. `/student/exams/:examId/start` - Exam instructions page (pre-attempt)
3. `/student/exams/:examId/attempt/:attemptId` - Active exam (LOCKED STATE)
4. `/student/exams/:examId/results/:attemptId` - Results page (post-submission)

**Exam Attempt Constraints**:
- Once started, student CANNOT navigate away without triggering anti-cheat events
- Browser back button disabled during active attempt
- Tab switching monitored and limited
- Fullscreen mode enforced (if configured)
- Time limits strictly enforced server-side

## 6. Data Flow & State Management

### How Data Moves from API → UI
```
1. User Action (e.g., click "Submit Assignment")
   ↓
2. React Component calls hook (e.g., useSubmitAssignment)
   ↓
3. React Query mutation executes API call via Axios
   ↓
4. Express route handler receives request
   ↓
5. Middleware validates auth, role, permissions
   ↓
6. Route calls service function (e.g., assignmentService.submit)
   ↓
7. Service executes business logic, validates data
   ↓
8. Service calls database via Drizzle ORM
   ↓
9. Database returns data
   ↓
10. Service transforms/enriches data
   ↓
11. Route sends JSON response
   ↓
12. React Query updates cache, triggers re-render
   ↓
13. UI updates with new data
```

### Global vs Local State
**Global State** (React Context):
- **AuthContext**: Current user, login/logout, role
- **ThemeContext**: Dark/light mode preference
- **StudentNotificationContext**: Real-time notifications for students

**Server State** (React Query):
- All API data (courses, assignments, exams, grades, etc.)
- Cached with automatic refetching and invalidation
- Mutations trigger cache updates

**Local State** (useState, useReducer):
- Form inputs, UI toggles, temporary data
- Component-specific state that doesn't need persistence

### Caching or Persistence Behavior
**React Query Cache**:
- Default stale time: 5 minutes (configurable per query)
- Automatic background refetching when data becomes stale
- Cache invalidation on mutations (e.g., after creating assignment, refetch assignment list)

**LocalStorage**:
- Theme preference persisted
- No sensitive data stored client-side

**Session Storage**:
- Not currently used

### Auth/Session Handling
**Authentication Flow**:
1. User submits credentials to `/api/auth/login`
2. Server validates credentials, generates JWT token
3. Token returned in response body (not HttpOnly cookie in current implementation)
4. Client stores token in memory (React Query cache)
5. Token included in `Authorization: Bearer <token>` header for all API requests
6. Server middleware validates token on protected routes

**Token Expiry**: 2 hours (configured in `server/src/middleware/auth.middleware.ts`)

**Logout**: Clears token from client state, redirects to login

## 7. API Integration

### How API Calls Are Structured
**Client-Side Pattern**:
```typescript
// In client/src/lib/api.ts or similar
export const apiClient = axios.create({
  baseURL: '/api',
  headers: {
    'Content-Type': 'application/json'
  }
});

// In React Query hook
const { data, isLoading } = useQuery({
  queryKey: ['courses', courseId],
  queryFn: () => apiClient.get(`/courses/${courseId}`).then(res => res.data)
});
```

**Server-Side Pattern**:
```typescript
// In server/src/api/{feature}.routes.ts
router.get('/:id', isAuthenticated, async (req, res) => {
  try {
    const data = await service.getData(req.params.id);
    res.json(data);
  } catch (error) {
    res.status(500).json({ message: 'Error' });
  }
});
```

### API Client Location
- **Client**: API calls made via Axios, configured in `client/src/lib/`
- **Server**: Route handlers in `server/src/api/`, organized by feature

### Error Handling Strategy
**Server-Side**:
- Try-catch blocks in all route handlers
- Consistent error response format: `{ message: string, errors?: object }`
- HTTP status codes: 400 (validation), 401 (auth), 403 (forbidden), 404 (not found), 500 (server error)
- Logging via console.error (production should use Winston or similar)

**Client-Side**:
- React Query handles loading/error states
- Toast notifications for user feedback (via `useToast` hook)
- Error boundaries for catastrophic failures (not fully implemented)

### Mock Data vs Real Data Rules
**Development**:
- Demo users available via `/api/auth/create-demo-users` endpoint
- AI chatbot has demo mode fallback when API keys not configured
- No mock data in production builds

**Production**:
- All data from PostgreSQL database
- No hardcoded mock data in components

## 8. Exam System Rules (CRITICAL)

### What Parts of Exams Are Mutable vs Immutable

**IMMUTABLE (Cannot Change After Students Start Attempting)**:
- Exam questions (text, options, correct answers)
- Question order (if shuffle disabled)
- Total points per question
- Passing score
- Anti-cheat settings (if attempts exist)

**MUTABLE (Can Change Before First Attempt)**:
- Exam title, description, instructions
- Scheduled start/end times (if no attempts yet)
- Status (draft → scheduled → active)

**PARTIALLY MUTABLE (Can Change With Caution)**:
- Time limit (can extend, but not reduce if attempts exist)
- Attempts allowed (can increase, but not decrease)
- Late submission settings (can enable, but not disable if late submissions exist)

### Exam Settings vs Exam Content Boundaries

**Exam Settings** (stored in `exams` table):
- Scheduling, timing, attempts, display options
- Anti-cheat configuration
- Retake policy
- **Can be modified** by teachers before exam becomes active

**Exam Content** (stored in `exam_questions` table):
- Question text, options, correct answers, points
- **MUST NOT be modified** after first student attempt
- To change content: Archive exam, create new version

### Safe vs Unsafe Changes

**SAFE Changes**:
- Creating new draft exam
- Adding questions to draft exam
- Updating exam settings before activation
- Changing status from draft → scheduled
- Extending time limits before first attempt
- Updating instructions/description

**UNSAFE Changes (Require Validation)**:
- Modifying questions after attempts exist → **FORBIDDEN**
- Changing correct answers after grading → **FORBIDDEN**
- Reducing time limit with active attempts → **FORBIDDEN**
- Disabling anti-cheat with flagged attempts → **REQUIRES REVIEW**
- Changing passing score after grading → **REQUIRES RE-GRADING**

**Implementation**:
```typescript
// In server/src/api/exam.routes.ts
// Validation before updates
const [existingExam] = await db.select().from(exams).where(eq(exams.id, examId));

if (existingExam.status === 'active' || existingExam.status === 'completed') {
  // Check if attempts exist
  const attempts = await db.select().from(examAttempts).where(eq(examAttempts.examId, examId));
  
  if (attempts.length > 0) {
    // Restrict modifications
    return res.status(400).json({ 
      message: 'Cannot modify exam with existing attempts' 
    });
  }
}
```

### Active Attempt Constraints

**During Active Attempt** (`exam_attempts.status === 'in_progress'`):
1. Student CANNOT:
   - Navigate away from exam page
   - Open new tabs (monitored)
   - Copy/paste (if disabled)
   - Right-click (if disabled)
   - Exit fullscreen (if required)
   - Pause timer (server-side enforcement)

2. System MUST:
   - Track all anti-cheat events
   - Enforce time limits server-side
   - Auto-submit on time expiry
   - Prevent multiple simultaneous attempts

3. Teacher CANNOT:
   - Modify exam questions
   - Change time limit
   - Delete exam
   - Change anti-cheat settings

**Implementation**: See `server/src/services/exam-attempt.service.ts` for attempt lifecycle management

## 9. Anti-Cheat & Integrity Logic (CRITICAL)

### What Signals Are Collected

**Browser Events** (Client-Side Detection):
- `tab_switch`: User switches browser tabs
- `window_blur`: Browser window loses focus
- `copy_paste`: Copy/paste attempted (if disabled)
- `right_click`: Right-click attempted (if disabled)
- `keyboard_shortcut`: Suspicious shortcuts (Ctrl+C, Ctrl+V, etc.)
- `devtools_open`: Developer tools opened
- `fullscreen_exit`: User exits fullscreen mode
- `multiple_monitors`: Multiple displays detected

**Biometric Events** (If Enabled):
- `face_not_detected`: Webcam shows no face
- `multiple_faces`: More than one person detected
- `no_face_visible`: Face obscured or turned away

**Pattern Analysis** (Server-Side Detection):
- `suspicious_pattern`: Unusual answer timing patterns
- `rapid_answers`: Answering too quickly
- `unusual_timing`: Inconsistent time per question

**Device/Network Events**:
- `unauthorized_app`: Suspicious applications detected
- `browser_extension_detected`: Extensions that could aid cheating
- `screen_share_detected`: Screen sharing software active

### Where Events Are Logged

**Database Table**: `anti_cheat_events`
```sql
CREATE TABLE anti_cheat_events (
  id TEXT PRIMARY KEY,
  attempt_id TEXT NOT NULL,  -- Links to exam_attempts
  event_type anti_cheat_event_type_enum NOT NULL,
  severity event_severity_enum NOT NULL,  -- low, medium, high, critical
  description TEXT,
  metadata JSONB,  -- Event-specific data
  detected_by VARCHAR(100),  -- 'browser_monitor', 'ai_model', 'proctor'
  timestamp TIMESTAMP NOT NULL,
  review_status review_status_enum DEFAULT 'pending'
);
```

**Logging Flow**:
1. Client detects event → Sends to `/api/anti-cheat/events`
2. Server validates, enriches with metadata
3. Service calculates severity based on event type
4. Event stored in database
5. Risk score updated in `anti_cheat_risk_scores` table
6. If threshold exceeded, attempt flagged for review

**Service**: `server/src/services/anti-cheat-monitor.service.ts`

### What Is Automated vs Reviewed by Humans

**Automated**:
- Event detection and logging
- Risk score calculation (0-100 scale)
- Automatic flagging when thresholds exceeded
- Auto-invalidation for critical violations (configurable)
- Student notifications for minor violations

**Human Review Required**:
- All attempts with `flaggedForReview === true`
- Risk score > 70 (high risk)
- Critical severity events (3+ occurrences)
- Disputed grades or violations
- Final verdict on flagged attempts

**Review Workflow**:
1. Teacher views flagged attempts at `/teacher/attempts/:attemptId/review`
2. Reviews anti-cheat events, risk scores, recordings (if enabled)
3. Makes decision: `cleared`, `warning`, `violation`, `invalidated`
4. Updates `anti_cheat_risk_scores.finalVerdict`
5. Student notified of decision

### What MUST NOT Be Changed Lightly

**CRITICAL - DO NOT MODIFY**:
1. **Event Severity Mappings** (`server/src/services/risk-scoring.service.ts`)
   - Changing severity levels affects all risk calculations
   - Could invalidate historical data
   - Requires re-scoring all existing attempts

2. **Risk Score Algorithms** (`server/src/services/risk-scoring.service.ts`)
   - Algorithm changes affect fairness and consistency
   - Must be versioned and documented
   - Requires A/B testing before deployment

3. **Auto-Invalidation Thresholds**
   - Too strict: False positives, student complaints
   - Too lenient: Cheating goes undetected
   - Must be configurable per exam, not hardcoded

4. **Event Detection Logic** (Client-side monitors)
   - False positives harm user experience
   - False negatives defeat purpose
   - Requires extensive testing across browsers/devices

**Safe to Modify**:
- UI for displaying events to teachers
- Notification messages to students
- Report generation logic
- Analytics dashboards

## 10. Mistake Tracking & Retakes

### How Mistakes Are Stored

**Database Table**: `mistake_pool`
```sql
CREATE TABLE mistake_pool (
  id TEXT PRIMARY KEY,
  student_id TEXT NOT NULL,
  attempt_id TEXT NOT NULL,
  answer_id TEXT NOT NULL,
  question_id TEXT NOT NULL,
  exam_id TEXT NOT NULL,
  mistake_type mistake_type_enum NOT NULL,  -- wrong_answer, partial_credit, timeout, skipped
  topic VARCHAR(255),
  subtopic VARCHAR(255),
  skill_tag VARCHAR(255),
  difficulty_level VARCHAR(50),
  student_answer JSONB,
  correct_answer JSONB,
  points_lost REAL,
  remediation_status remediation_status_enum DEFAULT 'not_started',
  included_in_retake BOOLEAN DEFAULT false,
  corrected_in_retake BOOLEAN,
  is_repeated_mistake BOOLEAN DEFAULT false,
  repetition_count INTEGER DEFAULT 1
);
```

**Population Flow**:
1. Student submits exam attempt
2. Grading service evaluates answers (`server/src/services/answer-evaluation.service.ts`)
3. Incorrect/partial answers identified
4. Mistake records created with metadata (topic, skill, difficulty)
5. Pattern analysis detects repeated mistakes
6. Remediation status tracked

### How Retakes Are Generated

**Adaptive Retake Generation** (`server/src/services/retake-exam-generator.service.ts`):

1. **Trigger**: Student fails exam OR teacher enables retakes
2. **Analysis**: System analyzes mistakes from `mistake_pool`
3. **Question Selection Strategy**:
   - Prioritize topics with most mistakes
   - Include similar questions (same topic/skill)
   - Adjust difficulty based on performance
   - Ensure coverage of all weak areas
4. **Retake Exam Creation**:
   - Stored in `mistake_retake_exams` table
   - Links to original exam and attempt
   - Contains subset of questions targeting mistakes
   - Configurable: adaptive difficulty, question count
5. **Availability**:
   - Delay period enforced (default 24 hours)
   - Expiration date set
   - Student notified when available

**Retake Configuration** (per exam):
- `retakeEnabled`: Boolean, default true
- `retakeDelay`: Hours before retake available, default 24
- `adaptiveRetake`: Boolean, use mistake-based questions, default true

### Learning vs Graded Flows

**Graded Exams** (Default):
- Scores recorded in gradebook
- Contribute to final grade
- Strict anti-cheat enforcement
- Limited attempts
- Mistakes tracked for retakes

**Learning/Practice Mode** (Not Fully Implemented):
- Unlimited attempts
- Immediate feedback
- Show correct answers
- No grade impact
- Relaxed anti-cheat

**Current Implementation**: All exams are graded. Practice mode is a future enhancement.

## 11. Permissions & Roles

### Role Definitions

**Database Enum**: `user_role_enum`
```sql
CREATE TYPE user_role AS ENUM ('student', 'teacher', 'admin', 'parent', 'proctor');
```

**Role Hierarchy**:
1. **Admin**: Full system access, can perform all actions
2. **Teacher**: Manage own courses, students, content, grading
3. **Parent**: View own children's data, communicate with teachers
4. **Student**: Access enrolled courses, submit work, take exams
5. **Proctor**: (Future) Monitor exams, review anti-cheat events

### Role-Based Access Rules

**Students Can**:
- View enrolled courses
- Submit assignments
- Take exams (if enrolled and scheduled)
- View own grades and progress
- Participate in study groups
- Message other students and teachers
- Use AI study buddy

**Students CANNOT**:
- View other students' submissions or grades
- Access teacher or admin pages
- Modify course content
- Create or grade assignments
- View anti-cheat events

**Teachers Can**:
- Create and manage own courses
- Upload lessons and materials
- Create assignments and exams
- Grade submissions
- View enrolled students' data
- Communicate with students and parents
- Review anti-cheat events for own exams
- Generate report cards

**Teachers CANNOT**:
- Access other teachers' courses (unless admin)
- Modify system-wide settings
- Create or delete users
- Access admin analytics

**Admins Can**:
- All teacher permissions
- Create/edit/delete users
- Manage all courses (any teacher)
- System-wide settings
- View all analytics
- Link parents to students
- Manage calendar events and announcements

**Parents Can**:
- View linked children's grades, assignments, attendance
- Communicate with children's teachers
- View course information
- Access progress reports

**Parents CANNOT**:
- Submit assignments on behalf of children
- Take exams
- Access other children's data
- Modify grades or content

### Where Enforcement Occurs

**Client-Side** (UI-level, not security):
- Route guards in `client/src/components/ProtectedRoute.tsx`
- Conditional rendering based on `user.role`
- Navigation menu filtering

**Server-Side** (Security-critical):
- **Middleware**: `server/src/middleware/auth.middleware.ts`
  - `isAuthenticated`: Validates JWT
  - `requireRole(['teacher', 'admin'])`: Enforces role
- **Route-Level**: Each route applies appropriate middleware
- **Service-Level**: Additional ownership checks (e.g., teacher owns course)

**Example**:
```typescript
// In server/src/api/course.routes.ts
router.post(
  '/courses',
  isAuthenticated,           // Must be logged in
  requireRole(['teacher', 'admin']),  // Must be teacher or admin
  async (req, res) => {
    // Additional check: verify teacher owns course for updates
    const course = await db.select().from(courses).where(eq(courses.id, courseId));
    if (course.teacherId !== req.user.id && req.user.role !== 'admin') {
      return res.status(403).json({ message: 'Forbidden' });
    }
    // Proceed with operation
  }
);
```

## 12. Known Risks & Fragile Areas

### Tightly Coupled Components

1. **Exam Attempt Lifecycle**:
   - **Files**: `server/src/services/exam-attempt.service.ts`, `server/src/api/exam-attempt.routes.ts`, `client/src/pages/student-exam-attempt.tsx`
   - **Risk**: Changes to attempt status flow can break grading, anti-cheat, or retakes
   - **Mitigation**: Always test full exam lifecycle (start → answer → submit → grade → retake)

2. **Anti-Cheat System**:
   - **Files**: `server/src/services/anti-cheat-monitor.service.ts`, `server/src/services/risk-scoring.service.ts`, client-side event monitors
   - **Risk**: Changes to event types or severity affect risk scores and flagging
   - **Mitigation**: Version risk scoring algorithms, maintain backward compatibility

3. **Mistake Tracking → Retake Generation**:
   - **Files**: `server/src/services/retake-exam-generator.service.ts`, `server/src/services/retake-submission.service.ts`
   - **Risk**: Changes to mistake classification affect retake question selection
   - **Mitigation**: Test retake generation with various mistake patterns

4. **Parent Portal Data Aggregation**:
   - **Files**: `server/src/services/parent.service.ts`, parent-* pages
   - **Risk**: Queries join multiple tables, performance-sensitive
   - **Mitigation**: Monitor query performance, use database indexes

### Areas Likely to Break If Modified Incorrectly

1. **Authentication Flow**:
   - **What**: JWT generation, validation, token refresh
   - **Why Fragile**: Breaks all protected routes if token format changes
   - **Safe Changes**: Add new claims to token, extend expiry
   - **Unsafe Changes**: Change signing algorithm, remove required claims

2. **Database Schema Migrations**:
   - **What**: Drizzle ORM migrations in `migrations/`
   - **Why Fragile**: Incorrect migrations can corrupt data or break app
   - **Safe Changes**: Add new tables/columns, create indexes
   - **Unsafe Changes**: Drop columns with data, change column types without migration

3. **Exam Question Grading Logic**:
   - **What**: `server/src/services/answer-evaluation.service.ts`
   - **Why Fragile**: Incorrect grading affects student scores, mistake tracking, retakes
   - **Safe Changes**: Add new question types, improve partial credit logic
   - **Unsafe Changes**: Change scoring algorithm without re-grading existing attempts

4. **WebSocket Message Handling**:
   - **What**: `server/src/websocket.ts`, real-time messaging
   - **Why Fragile**: Message format changes break client-server communication
   - **Safe Changes**: Add new message types, add optional fields
   - **Unsafe Changes**: Remove required fields, change event names

### Performance-Sensitive Logic

1. **Parent Dashboard Queries**:
   - **Location**: `server/src/services/parent.service.ts`
   - **Issue**: Joins multiple tables (students, courses, assignments, grades, attendance)
   - **Optimization**: Use database indexes, limit date ranges, paginate results

2. **Analytics Calculations**:
   - **Location**: `server/src/api/analytics.routes.ts`
   - **Issue**: Aggregations over large datasets (all attempts, all students)
   - **Optimization**: Cache results, use database views, background jobs for heavy calculations

3. **Mistake Pattern Analysis**:
   - **Location**: `server/src/services/retake-exam-generator.service.ts`
   - **Issue**: Analyzes all mistakes for a student across all exams
   - **Optimization**: Limit to recent exams, use indexes on topic/skill fields

4. **Real-Time Messaging**:
   - **Location**: `server/src/websocket.ts`
   - **Issue**: Broadcasting messages to all group members
   - **Optimization**: Use rooms/channels, limit message history, paginate

### What AI Agents Are Allowed to Change

**SAFE for AI Modification**:
1. **UI Components**: Add/modify React components for new features
2. **New API Endpoints**: Create new routes following existing patterns
3. **Database Schema Additions**: Add new tables/columns via Drizzle migrations

6. **Error Messages**: Improve user-facing error messages
7. **Documentation**: Update comments, README files
8. **New Features**: Implement new features that don't affect core systems

**REQUIRES HUMAN REVIEW**:
1. **Authentication/Authorization**: Changes to JWT, role checks, permissions
2. **Exam Grading Logic**: Modifications to scoring algorithms
3. **Anti-Cheat Algorithms**: Changes to risk scoring, event severity
4. **Database Migrations**: Schema changes affecting existing data
5. **Performance Optimizations**: Query changes, caching strategies
6. **Security Updates**: CORS, rate limiting, input sanitization

### What AI Agents Must NEVER Change

**FORBIDDEN for AI Modification**:
1. **Exam Questions After Attempts Exist**: Violates academic integrity
2. **Grading Algorithms Without Re-Grading**: Creates inconsistent scores
3. **Anti-Cheat Event Types**: Breaks historical event analysis
4. **User Roles/Permissions Without Audit**: Security risk
5. **Database Schema Without Migrations**: Data corruption risk
6. **Production Environment Variables**: Deployment risk
7. **Cryptographic Keys/Secrets**: Security breach risk

### Required Validation Before Making Changes

**Pre-Modification Checklist**:
1. **Check for Active Exams**: Query `exam_attempts` for in-progress attempts
2. **Verify No Data Loss**: Ensure migrations preserve existing data
3. **Test Role-Based Access**: Verify permissions still work for all roles
4. **Run Existing Tests**: (Note: Test suite not fully implemented)
5. **Check Dependencies**: Ensure changes don't break dependent features
6. **Review Error Handling**: Ensure graceful degradation on failures

**Validation Queries**:

```sql
-- Check for active exam attempts
SELECT COUNT(*) FROM exam_attempts WHERE status = 'in_progress';

-- Check for exams with attempts
SELECT exam_id, COUNT(*) as attempt_count

SELECT COUNT(*) FROM exam_attempts WHERE flagged_for_review = true;
```

### Testing Expectations

**Expected Testing**:

2. **Role Testing**: Test as student, teacher, admin, parent

## 14. Conventions & Standards

### Naming Conventions

**Files**:
- React components: PascalCase (e.g., `StudentDashboard.tsx`)
- Utility files: kebab-case (e.g., `api-client.ts`)
- Route files: kebab-case with `.routes.ts` suffix (e.g., `exam.routes.ts`)
- Service files: kebab-case with `.service.ts` suffix (e.g., `exam-attempt.service.ts`)

**Variables/Functions**:
- camelCase for variables and functions (e.g., `examAttempt`, `calculateScore`)
- PascalCase for React components and types (e.g., `ExamAttempt`, `UserRole`)
- UPPER_SNAKE_CASE for constants (e.g., `MAX_ATTEMPTS`, `DEFAULT_TIME_LIMIT`)

**Database**:
- snake_case for table and column names (e.g., `exam_attempts`, `student_id`)
- Plural for table names (e.g., `exams`, `users`)
- Enums suffixed with `_enum` (e.g., `user_role_enum`)

**Routes**:
- Kebab-case for URL paths (e.g., `/student/exam-attempt`)
- Plural for collections (e.g., `/api/exams`)
- Singular for single resources (e.g., `/api/exams/:id`)

### File Organization Rules

**Client Structure**:
```
client/src/
├── pages/           # One file per page, role-prefixed
├── components/      # Reusable components, organized by feature
├── hooks/           # Custom React hooks
├── contexts/        # React Context providers
├── lib/             # Utilities, API client, constants
└── styles/          # Global styles, Tailwind config
```

**Server Structure**:
```
server/src/
├── api/             # Route handlers, one file per feature
├── services/        # Business logic, one file per domain
├── db/              # Database schema, connection
├── middleware/      # Express middleware
└── utils/           # Utility functions
```

### Commenting Philosophy

**When to Comment**:
- Complex business logic (e.g., risk scoring algorithms)
- Non-obvious workarounds or hacks
- Security-critical sections
- API endpoint documentation (request/response format)
- Database schema relationships

**When NOT to Comment**:
- Self-explanatory code
- Obvious variable names
- Standard patterns (e.g., React hooks)

**Style**:
```typescript
// Good: Explains WHY
// Use bcrypt with 12 rounds for security/performance balance
const saltRounds = 12;

// Bad: Explains WHAT (obvious from code)
// Hash the password
const hash = await bcrypt.hash(password, saltRounds);
```

### Error Handling Conventions

**Server-Side**:
```typescript
try {
  const result = await service.doSomething();
  res.json(result);
} catch (error) {
  console.error('Error in doSomething:', error);
  res.status(500).json({ 
    message: 'Failed to do something',
    // Include error details in development only
    error: process.env.NODE_ENV === 'development' ? error.message : undefined
  });
}
```

**Client-Side**:
```typescript
const { data, error, isLoading } = useQuery({
  queryKey: ['resource'],
  queryFn: fetchResource,
  onError: (error) => {
    toast({
      title: 'Error',
      description: error.message,
      variant: 'destructive'
    });
  }
});
```
